# Enrollment Readiness Flags

## What this document is

The complete specification of every Enrollment Readiness Flag LIFT raises, including the exact condition that makes each flag appear, the data source behind it, severity thresholds, and auto-resolution rules.

Admins and schools read this document to answer the question: **"why was this flag raised?"** Every flag in LIFT maps to a row in this document. If a parent, attorney, or board member asks what a flag means, the answer is here — no tribal knowledge, no guessing.

## What Enrollment Readiness Flags are NOT

- **Not predictions.** A flag does not say a candidate will withdraw, will not enroll, or is "at risk." A flag says that a specific observable condition currently holds — nothing about what will happen.
- **Not scores.** Flags are not ranked or weighted. Each one is independent. A candidate with three flags is not "three times more concerning" than a candidate with one.
- **Not probabilities.** There is no percentage, likelihood, or statistical confidence attached to a flag.
- **Not generated by a model.** Flags are produced by deterministic code that checks real database fields. No AI, no inference, no training data.

Year-two work may introduce a predictive model trained against resolved flag history and actual outcomes. That will be a separate, explicitly-labeled feature. Everything in this document describes the deterministic observation layer shipped in Stage 4 of LIFT V2.

## How flags are evaluated

A daily cron at 10:00 UTC walks every candidate in an eligible status (`completed`, `flagged`, `reviewed`, `admitted`, `waitlisted`, `offered`) and runs each of the seven flag definitions against the candidate's current state.

For each (candidate, flag) pair the evaluator does one of:

- **Raise** (insert new active flag) if the condition holds and no active flag of this type exists.
- **Refresh** (update `last_observed_at`) if the condition holds and an active flag already exists at the same severity. No audit row is written for refreshes — otherwise a long-lived flag would flood the audit log with daily noise.
- **Escalate** (update active flag to higher severity, clear any snooze, write audit row) if the condition escalates from `advisory` to `notable`.
- **Auto-resolve** (resolve active flag with `resolution_type = auto_condition_cleared`) if the condition no longer holds.
- **Respect snooze** (no-op) if the flag is resolved with `snooze_until` in the future AND the severity has not escalated.
- **Re-raise from snooze** (insert new active flag) if snooze expired OR severity escalated beyond what was resolved.

## Resolution

Any `school_admin` or `platform_admin` can manually resolve an active flag via the drawer on `/school/flags`, `/school/briefing`, the candidate detail page, or the committee deliberation card. Manual resolution requires a `resolved_reason` (logged to audit) and an optional `snooze_days` override (1-90, default 30).

If the underlying condition persists past the snooze window, the flag re-raises. If the condition worsens during the snooze window (e.g. a `consent_not_captured` advisory is still unresolved at day 8 and escalates to notable), the flag re-raises immediately regardless of snooze state.

## Flag Catalog

### 1. `consent_not_captured`

**What we observed:** The candidate's status is `admitted`, `waitlisted`, or `offered`, but we have not recorded both the `student_consent` and `guardian_consent` events in the consent log.

**Raise condition (prose):** Candidate has an active admit-adjacent status AND at least one expected consent type is missing from `consent_events` for this candidate.

**Raise condition (SQL-style):**
```
candidates.status IN ('admitted','waitlisted','offered')
  AND final_recommendations exists for this candidate
  AND NOT (EXISTS consent_events WHERE consent_type = 'student_consent')
  OR  NOT (EXISTS consent_events WHERE consent_type = 'guardian_consent')
```

**Severity thresholds:**
- `notable` when more than 7 days have passed since the admit decision
- `advisory` when 7 days or fewer have passed since the admit decision

**Auto-resolve conditions:**
- Both expected consent types appear in `consent_events` → `auto_condition_cleared`
- CORE handoff completes for this candidate → `auto_core_handoff`

**Data sources:** `candidates.status`, `final_recommendations.decided_at`, `consent_events`.

**Example `computed_from` payload:**
```json
{
  "expected_consent_types": ["student_consent", "guardian_consent"],
  "found_consent_types": ["student_consent"],
  "missing_consent_types": ["guardian_consent"],
  "days_since_admit": 11,
  "admit_decided_at": "2026-04-10T14:03:00Z"
}
```

**What this flag does NOT mean:** It does not mean the family will not enroll. It does not mean consent has been refused. It means we have not observed the record of consent. The consent may exist elsewhere (on paper, in email, in CORE after handoff) — this flag simply states that LIFT's consent_events table does not have a matching row yet.

**Known limitation:** The expected consent types list is hardcoded to `['student_consent', 'guardian_consent']` in Stage 4. If a pilot school uses different consent vocabularies, they will see this flag raise when it shouldn't. Documented in the Stage 4 ship report as a year-two generalization candidate.

### 2. `invite_expired_unopened`

**What we observed:** The candidate's most recent invite has an `expires_at` in the past, and `opened_at` is NULL.

**Raise condition (prose):** The most recent invite record for this candidate has passed its expiration date without being opened.

**Raise condition (SQL-style):**
```
invites.expires_at < NOW()
  AND invites.opened_at IS NULL
```

**Severity thresholds:** Always `notable`. (There is no "partially expired" — either the link is past its expiry or it isn't.)

**Auto-resolve conditions:**
- The invite is opened (`opened_at` becomes non-NULL) → `auto_condition_cleared`
- A newer invite replaces this one and is opened → `auto_condition_cleared`

**Data sources:** `invites` (most recent by `created_at`).

**Example `computed_from` payload:**
```json
{
  "invite_id": "a18f3e9c-4201-4f8a-...",
  "sent_at": "2026-03-21T09:00:00Z",
  "expires_at": "2026-03-28T09:00:00Z",
  "opened_at_was_null": true,
  "status": "pending"
}
```

**What this flag does NOT mean:** It does not mean the family is uninterested. They may have received the email, intended to open it, and not gotten to it in time. The observation is simply that our invite system recorded the email sent but never recorded a view event.

### 3. `assessment_abandoned`

**What we observed:** The candidate's most recent assessment session is either explicitly marked `abandoned` OR has had no activity for more than 7 days while in a non-completed status.

**Raise condition (prose):** The session is either explicitly abandoned OR stalled (in progress / paused / not started) with no activity recorded in more than 7 days.

**Raise condition (SQL-style):**
```
sessions.status = 'abandoned'
  OR (sessions.status != 'completed'
      AND sessions.last_activity_at < NOW() - interval '7 days')
```

**Severity thresholds:** Always `notable`. Abandoned or long-stalled assessments need direct admissions outreach regardless of how long.

**Auto-resolve conditions:**
- Session resumes and reaches `completed` status → `auto_condition_cleared`
- Session resumes with new recent activity AND status is no longer abandoned → `auto_condition_cleared`

**Data sources:** `sessions` (most recent by `created_at`).

**Example `computed_from` payload:**
```json
{
  "session_id": "c7d2...",
  "status": "in_progress",
  "completion_pct": 42,
  "last_activity_at": "2026-04-01T15:40:00Z",
  "days_since_activity": 20
}
```

**What this flag does NOT mean:** It does not mean the candidate gave up or is disengaged. Students pause assessments for many reasons — illness, schedule conflicts, device issues. The flag just says the session is stalled.

### 4. `low_completion`

**What we observed:** The candidate completed their assessment session, but finished fewer than 75% of the tasks.

**Raise condition (prose):** Session status is `completed`, and `completion_pct` is below 75.

**Raise condition (SQL-style):**
```
sessions.status = 'completed'
  AND sessions.completion_pct < 75
```

**Severity thresholds:** Always `advisory`. A partially-completed assessment is context for the committee, not an urgent action item.

**Auto-resolve conditions:**
- Session's `completion_pct` rises to 75 or above (unusual; would require a re-opened session) → `auto_condition_cleared`

**Data sources:** `sessions.status`, `sessions.completion_pct`.

**Example `computed_from` payload:**
```json
{
  "session_id": "c7d2...",
  "completion_pct": 58,
  "completed_at": "2026-04-10T14:00:00Z"
}
```

**What this flag does NOT mean:** It does not mean the candidate is less capable or less engaged than peers. Completion percentage varies with session length, platform familiarity, and individual pace. The flag exists so committees have this context visible when deliberating.

### 5. `late_cycle_admit`

**What we observed:** An admit or waitlist decision was recorded within 7 days of the application cycle's close date.

**Raise condition (prose):** The candidate's most recent `final_recommendations.decision` is `admit` or `waitlist`, and `decided_at` is within 7 days of the cycle's `closes_at`.

**Raise condition (SQL-style):**
```
final_recommendations.decision IN ('admit', 'waitlist')
  AND final_recommendations.decided_at > application_cycles.closes_at - interval '7 days'
  AND final_recommendations.decided_at <= application_cycles.closes_at
```

**Severity thresholds:** Always `advisory`. Late-cycle admits leave less time for family response, but don't indicate anything wrong.

**Auto-resolve conditions:**
- The candidate's cycle is replaced with a new cycle whose close date is far in the future → `auto_condition_cleared`

**Data sources:** `final_recommendations`, `application_cycles.closes_at`.

**Example `computed_from` payload:**
```json
{
  "decided_at": "2026-04-18T16:00:00Z",
  "cycle_closes_at": "2026-04-22T23:59:59Z",
  "days_before_close": 4,
  "decision": "admit"
}
```

**What this flag does NOT mean:** It does not mean the committee made a hasty or poor decision. Sometimes the best candidates apply late, or the committee defers until later rounds for good reason. The flag exists so admissions teams notice compressed response windows and can plan follow-up accordingly.

### 6. `post_admit_silence`

**What we observed:** The candidate was admitted, and we have observed no activity — consent events, invite opens, status changes, interviewer assignments, or application-data updates — within the tenant's configured silence window (default 14 days, range 1-90, per `tenant_settings.post_admit_silence_days`).

**Raise condition (prose):** Candidate's status is `admitted`, more than N days have passed since the admit decision, and no activity signal has been recorded in the last N days.

**Raise condition (SQL-style):**
```
candidates.status = 'admitted'
  AND NOW() - final_recommendations.decided_at > interval 'N days'
  AND NOT EXISTS (any activity signal in the last N days, where activity is:
    consent_events.consented_at,
    invites.opened_at,
    candidate_status_history.changed_at,
    candidate_assignments.created_at,
    candidate_application_data.updated_at)
```

**Severity thresholds:** Always `notable`. Silence past the configured window is a meaningful absence of engagement signal.

**Auto-resolve conditions:**
- Any activity signal appears in the last N days → `auto_condition_cleared`
- CORE handoff completes → indirectly resolves via `consent_not_captured` chain, but `post_admit_silence` itself clears when new activity is observed

**Data sources:** `candidates.status`, `final_recommendations`, `consent_events`, `invites`, `candidate_status_history`, `candidate_assignments`, `candidate_application_data`, `tenant_settings.post_admit_silence_days`.

**Example `computed_from` payload:**
```json
{
  "admit_date": "2026-03-28T10:00:00Z",
  "most_recent_activity_at": "2026-03-28T10:00:00Z",
  "silence_days": 24,
  "threshold_days": 14
}
```

**What this flag does NOT mean:** It does not mean the family has decided not to enroll. It means we have not observed the signals that typically accompany active enrollment preparation. There may be activity happening in systems LIFT doesn't see (phone calls, in-person meetings).

### 7. `interviewer_unresponsive`

**What we observed:** A candidate has been assigned to an interviewer more than 14 days ago, and no interview rubric has been submitted.

**Raise condition (prose):** A `candidate_assignments` row with `assignment_type IN ('interview', 'both')` exists more than 14 days ago, and no `interview_rubric_submissions` row exists for this candidate.

**Raise condition (SQL-style):**
```
candidate_assignments.assignment_type IN ('interview', 'both')
  AND NOW() - candidate_assignments.created_at > interval '14 days'
  AND NOT EXISTS (SELECT 1 FROM interview_rubric_submissions WHERE candidate_id = candidate.id)
```

**Severity thresholds:** Always `advisory`. Delayed interview rubrics need attention but aren't urgent in the same way abandoned assessments are.

**Auto-resolve conditions:**
- An `interview_rubric_submissions` row is created for the candidate → `auto_condition_cleared`
- The assignment is withdrawn or completed → `auto_condition_cleared`

**Data sources:** `candidate_assignments`, `interview_rubric_submissions`.

**Example `computed_from` payload:**
```json
{
  "assignment_id": "b92a...",
  "assigned_to": "user-id-of-interviewer",
  "assigned_at": "2026-04-01T10:00:00Z",
  "days_since_assignment": 20
}
```

**What this flag does NOT mean:** It does not mean the interviewer is unreliable or that the candidate will be rejected. Interviewers may be waiting on scheduling, handling a sick day, or processing a complex case. The flag is a queue-management signal for the admissions admin, not a performance judgment on the interviewer. It is intentionally not surfaced to interviewers themselves.

## Observation-vs-prediction discipline

Every flag in this catalog is defined in terms of what we have *observed* in the data. No flag asserts anything about what will happen next. When describing flags to families, board members, or attorneys, use the same framing:

> "Our admissions system tracked that [specific observable condition] held for this candidate. This is one of several factual observations the team reviews. It is not a prediction, not a score, and not a probability."

If you notice language anywhere in the LIFT product, audit logs, or documentation that describes flags in predictive terms — "at risk," "likely to withdraw," "predicted to," "yield risk," "withdrawal risk" — flag it as a regression. The deterministic observation model only defends itself if the language discipline holds. See `memory/feedback_observation_language.md` for the internal discipline note.
